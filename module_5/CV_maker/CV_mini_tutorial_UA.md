# Міні-туторіал: як перетворити звичайне CV на сильне професійне резюме

## Мета туторіалу

Цей міні-туторіал показує, як покращити CV не через “красиві слова”, а через правильну професійну логіку.

Головна ідея:

```text
Слабке CV описує, що людина робила.
Сильне CV показує, яку цінність людина створила.
```

CV має відповідати на чотири питання:

```text
1. Хто я професійно?
2. Які проблеми я вмію вирішувати?
3. Якими інструментами я це роблю?
4. Який результат або цінність це створює?
```

---

## 1. Professional Summary: не біографія, а elevator pitch

### Типова помилка

У CV часто пишуть кілька коротких речень, які повторюють одну й ту саму думку:

```text
I am a Python GIS Engineer.
I am a hydrologist.
I am a researcher.
I develop geospatial pipelines.
I work with satellite data.
```

Проблема: читач бачить перелік ролей, але не бачить цілісного професійного образу.

---

### Кращий підхід

Professional Summary має бути коротким elevator pitch.

Формула:

```text
роль + досвід + спеціалізація + доказ сили + ключові технології
```

---

### Приклад

Слабше:

```text
Python GIS Engineer and hydrologist developing geospatial data pipelines for environmental monitoring, flood-risk assessment and spatial analytics.
```

Сильніше:

```text
Python GIS Engineer & Hydrologist with 15+ years of domain experience and a focus on AI-driven environmental analytics, floodplain modeling and DEM validation. Founder of GeoHydroAI and recipient of the DBU Environmental Innovation Grant, with proven expertise in building automated geospatial data pipelines using Python, PostGIS, DuckDB and remote sensing data.
```

---

### Чому це краще

У сильному варіанті одразу видно:

- професійну роль;
- доменний досвід;
- спеціалізацію;
- унікальність кандидата;
- технологічний стек;
- зовнішні докази сили: проєкт, грант, публікації.

---

## 2. Experience: від процесу до результату

### Типова помилка

Багато CV описують тільки процес:

```text
Developing Python pipelines...
Implementing spatial analysis workflows...
Building flood forecasting models...
Validating DEMs...
```

Це не погано, але цього недостатньо.

Рекрутер або науковий керівник хоче зрозуміти:

```text
І що з цього?
Яку проблему це вирішило?
Який результат це дало?
```

---

### Формула сильного bullet point

```text
Action verb + object + tools/data/methods + value/result
```

Або українською:

```text
Сильне дієслово + що зроблено + чим/на яких даних + для чого/яка цінність
```

---

### Приклад 1

Слабше:

```text
Developing Python pipelines for processing large geospatial datasets.
```

Сильніше:

```text
Engineered automated Python pipelines for processing Sentinel-1/2 imagery and global DEMs, improving reproducibility of flood-risk and DEM-validation workflows.
```

---

### Розбір прикладу

```text
Engineered
```

Сильне дієслово. Воно показує не просто участь, а створення системи.

```text
automated Python pipelines
```

Об'єкт роботи. Видно, що це не разовий скрипт, а автоматизований процес.

```text
Sentinel-1/2 imagery and global DEMs
```

Дані. Це додає технічну конкретику.

```text
improving reproducibility of flood-risk and DEM-validation workflows
```

Цінність. Видно, навіщо це було зроблено.

---

## 3. Сильні дієслова для CV

Замість слабких або загальних дієслів:

```text
Worked on...
Helped with...
Was responsible for...
Participated in...
```

краще використовувати:

```text
Engineered
Built
Designed
Developed
Automated
Validated
Integrated
Optimized
Implemented
Led
Created
Analyzed
Deployed
Structured
Maintained
```

---

### Приклад

Слабше:

```text
Participated in DEM validation.
```

Сильніше:

```text
Built geospatial validation workflows combining ICESat-2 LiDAR altimetry, xDEM, GDAL, GeoPandas and PostGIS to assess DEM accuracy in mountainous floodplain environments.
```

## 4. Метрики: додавати тільки чесні числа

Метрики дуже підсилюють CV, але їх не можна вигадувати.

Не варто писати:

```text
Reduced processing time by 40%.
Trained 100 students.
Processed 1 TB of data.
```

якщо цих чисел немає або вони не перевірені.

---

### Які метрики можна додати пізніше

Для GIS / Data Engineering:

```text
кількість оброблених DEM
кількість Sentinel scenes
площа території аналізу
кількість raster tiles
кількість точок ICESat-2
обсяг даних
час обробки до/після автоматизації
```

Для викладання:

```text
кількість студентів
кількість груп
кількість створених домашніх завдань
кількість навчальних модулів
```

Для наукової роботи:

```text
кількість публікацій
кількість цитувань
кількість конференцій
кількість проєктів
кількість dataset / model outputs
```

---

## 5. Publications / Projects: найсильніше — вище

У науковому, research-oriented або technical CV публікації й проєкти потрібно розміщувати стратегічно.

Головне правило:

```text
Найсильніше, найрелевантніше і найближче до цільової позиції має бути вище.
```

Це стосується не тільки публікацій, а й проєктів.

Якщо публікація або проєкт напряму підтримує твою професійну нішу, його варто поставити першим.

---

### Чому порядок важливий

Рекрутер, науковий керівник або грантовий reviewer не читає CV як книгу від початку до кінця.

Зазвичай він швидко сканує:

```text
хто кандидат
яка спеціалізація
які ключові проєкти
які докази експертності
чи є релевантні публікації
```

Тому перші пункти в розділах `Projects` і `Publications` мають одразу підсилювати твоє позиціонування.

---

### Приклад для research-oriented CV

Якщо професійна ніша така:

```text
Python GIS Engineer + Hydrologist + DEM validation + floodplain modeling
```

то дуже сильним доказом буде публікація:

```text
Validation of global elevation models using ICESat-2 LiDAR data for floodplain modeling in the Ukrainian Carpathians
```

Її логічно поставити першою, бо вона прямо підтверджує ключову експертизу кандидата:

```text
DEM validation
ICESat-2 LiDAR
floodplain modeling
Ukrainian Carpathians
hydrology + geospatial analysis
```

---

### Приклад для project-oriented CV

Якщо кандидат подається на позицію:

```text
GIS / Data Engineer
```

то першим проєктом краще поставити не навчальний репозиторій, а найбільш прикладний технічний проєкт.

Сильніше:

```text
GeoHydroAI Platform
Sentinel-1 Flood Detection Pipeline
Python Course Repository
```

Слабше:

```text
Python Course Repository
GeoHydroAI Platform
Sentinel-1 Flood Detection Pipeline
```

Чому?

Бо для GIS/Data Engineering позиції важливіше одразу показати:

```text
geospatial platform
satellite data
flood-risk assessment
Python pipelines
PostGIS / DuckDB / Dash
remote sensing
```

А навчальний репозиторій теж корисний, але він менш релевантний для основної технічної ролі.

---

### Як вибирати порядок проєктів

Постав собі питання:

```text
1. Який проєкт найкраще показує мою цільову роль?
2. Який проєкт найбільш технічно складний?
3. Який проєкт має реальний вплив або практичне застосування?
4. Який проєкт найкраще поєднує мої ключові технології?
5. Який проєкт найкраще підтверджує мій Professional Summary?
```

Першим має бути той проєкт, який дає найсильнішу відповідь на ці питання.

---

### Як вибирати порядок публікацій

Для публікацій логіка схожа.

Першою краще ставити не обов'язково найстарішу або навіть формально “найвідомішу”, а ту, яка найкраще підтримує твою поточну професійну ціль.

Наприклад:

```text
1. Найбільш релевантна до позиції публікація
2. Найновіша сильна публікація
3. Публікація у найсильнішому журналі
4. Інші публікації
```

Для academic CV іноді порядок може бути хронологічний або reverse chronological, але навіть там найважливіші публікації можна виділити окремо як `Selected Publications`.

---

### Формула сильного опису проєкту

Проєкт у CV має описуватися не просто як назва.

Краще використовувати формулу:

```text
назва проєкту + проблема + рішення + технології + цінність
```

Приклад:

```text
GeoHydroAI Platform — open geospatial platform integrating satellite data, DEM validation, hydrological models and machine learning for flood-risk assessment, environmental monitoring and landscape-restoration analytics.
```

Тут видно:

```text
що це: geospatial platform
які дані: satellite data, DEM
які методи: hydrological models, machine learning
для чого: flood-risk assessment, environmental monitoring
```

---

### Головний принцип

Публікації й проєкти — це докази.

Вони мають не просто показувати, що кандидат “щось робив”, а підтверджувати центральну професійну тезу CV.


### Правильний формат публікації

```html
<p>
<strong><a href="https://doi.org/10.24028/gj.v48i2.346831" target="_blank">
Validation of global elevation models using ICESat-2 LiDAR data for floodplain modeling in the Ukrainian Carpathians</a></strong><br>
Authors: V. V. Nikoriak, V. V. Osypov, N. M. Osadcha<br>
Journal: Geofizicheskiy Zhurnal, 48(2), 2026<br>
DOI: <a href="https://doi.org/10.24028/gj.v48i2.346831" target="_blank">10.24028/gj.v48i2.346831</a>
</p>
```

## 7. Languages: простий і стандартний формат

### Слабше

```text
English – Intermediate (B2 reading, strong listening)
```

Проблема: звучить неоднорідно й може заплутати ATS або рекрутера.

---

### Краще

```text
English – Upper-Intermediate (B2)
```

На співбесіді вже можна пояснити, що reading/listening сильніші, а speaking/writing розвиваються.

## 8. Page breaks у PDF: не розривати тільки логічний заголовок

Для HTML → PDF важливо, щоб заголовок місця роботи не відривався від посади й дат.

Поганий варіант:

```html
<section class="experience-item keep-together">
    <h3>Organization</h3>
    <p>Role, dates</p>
    <ul>
        <li>Long bullet point...</li>
    </ul>
</section>
```

Проблема: якщо весь блок заборонено розривати, PDF може залишати багато порожнього місця або зменшувати масштаб.

---

### Кращий варіант

```html
<section class="experience-item">

    <div class="experience-header keep-together">
        <h3>Organization</h3>
        <p>
            <strong>Role</strong><br>
            Dates<br>
            Location
        </p>
    </div>

    <ul>
        <li>Bullet points can continue on the next page.</li>
    </ul>

</section>
```

---

### CSS

```css
.keep-together {
    page-break-inside: avoid;
    break-inside: avoid;
}

.experience-item {
    margin-bottom: 20px;
}

.experience-header {
    page-break-inside: avoid;
    break-inside: avoid;
}

h2,
h3 {
    page-break-after: avoid;
    break-after: avoid;
}
```

---

### Логіка

```text
Не розривати:
назва організації + посада + дата

Можна розривати:
довгий список bullet points
```

## 9. ATS-оптимізація: простий дизайн читається краще

Сильне CV має бути не тільки красивим для людини, а й зрозумілим для ATS-систем.

ATS — це Applicant Tracking System, тобто система, яка автоматично читає CV, витягує з нього ключову інформацію і порівнює її з вимогами вакансії.

---

### Типова помилка

Кандидати часто намагаються зробити CV візуально “сучасним”:

```text
дві колонки
прогрес-бари навичок
іконки замість тексту
інфографіка
складні таблиці
текстові блоки збоку
нестандартні шрифти
```

Для людини це може виглядати красиво, але ATS-парсер може прочитати такий файл неправильно.

---

### Чому це проблема

ATS може:

```text
об'єднати текст із різних колонок
прочитати блоки в неправильному порядку
пропустити частину навичок
не розпізнати назви розділів
не зчитати текст з іконок або графіки
зламати структуру досвіду роботи
```

Наприклад, якщо CV має дві колонки, система може прочитати рядки так:

```text
Python 2022 – Present PostGIS Ukrainian Hydrometeorological Institute Docker GeoPandas
```

Для людини це були окремі блоки, але для машини це може перетворитися на хаотичний текст.

---

### Кращий підхід

Для ATS-friendly CV краще використовувати просту структуру:

```text
одна колонка
чіткі заголовки
звичайний текст
списки bullet points
стандартні назви розділів
простий HTML / PDF layout
```

Рекомендовані назви розділів:

```text
Professional Summary
Technical Skills
Selected Projects
Professional Experience
Education
Publications
Certifications
Languages
```

---

### Важливий принцип

```text
CV має бути красивим, але не ціною втрати машинної читабельності.
```

Для технічного CV краще мати простий, чистий і добре структурований дизайн, ніж складну інфографіку, яку ATS прочитає неправильно.

---

## 10. Дзеркалення ключових слів із вакансії

CV потрібно адаптувати під конкретну вакансію.

Один із найсильніших прийомів — дзеркалення ключових слів.

---

### Що таке дзеркалення

Дзеркалення означає, що якщо у вакансії використовується конкретне формулювання, варто перевірити, чи є таке саме або дуже близьке формулювання у твоєму CV.

---

### Приклад

У вакансії написано:

```text
spatial data processing
```

А в CV написано:

```text
GIS workflows
```

Це схожі поняття, але ATS може не завжди зрозуміти, що вони означають одне й те саме.

Краще адаптувати bullet point так, щоб там було формулювання з вакансії:

```text
Built automated spatial data processing workflows using Python, GeoPandas and PostGIS for flood-risk and environmental analysis.
```

---

### Інший приклад

У вакансії написано:

```text
remote sensing data pipelines
```

Слабше:

```text
Worked with satellite data.
```

Сильніше:

```text
Engineered remote sensing data pipelines for processing Sentinel-1/2 imagery and generating flood-risk analytics.
```

---

### Ще один приклад

У вакансії написано:

```text
PostGIS spatial database
```

Слабше:

```text
Used databases for geospatial data.
```

Сильніше:

```text
Designed PostGIS spatial database workflows for storing, indexing and querying large geospatial datasets.
```

---

### Як адаптувати CV під вакансію

Перед відправкою CV зроби короткий аналіз вакансії.

Випиши ключові слова:

```text
spatial data processing
remote sensing
PostGIS
Python pipelines
ETL
geospatial analytics
data validation
flood risk
machine learning
dashboard
```

Потім перевір, чи є ці слова у твоєму CV:

```text
Professional Summary
Technical Skills
Selected Projects
Professional Experience
```

---

### Важливо

Не треба бездумно вставляти ключові слова списком.

Погано:

```text
Python, PostGIS, remote sensing, spatial data processing, ETL, GIS, dashboard, machine learning.
```

Краще:

```text
Built automated spatial data processing pipelines using Python, GeoPandas and PostGIS to support remote sensing-based flood-risk assessment.
```

Тут ключові слова вбудовані в реальне професійне досягнення.

---

## 11. Правило адаптації CV під вакансію

Перед кожною подачею CV постав собі три питання:

```text
1. Які 5–10 ключових слів повторюються у вакансії?
2. Чи є ці формулювання в моєму CV?
3. Чи вони вбудовані в реальні досягнення, а не просто додані списком?
```

---

### Міні-приклад адаптації

Вакансія:

```text
We are looking for a GIS Data Engineer with experience in spatial data processing, PostGIS, Python pipelines and remote sensing datasets.
```

CV має містити щось подібне:

```text
Engineered Python-based spatial data processing pipelines using GeoPandas, PostGIS and remote sensing datasets for flood-risk and environmental analytics.
```

Тут CV “дзеркалить” вакансію, але не копіює її механічно. Воно показує, що кандидат реально має потрібний досвід.

---

## 12. Чекліст сильного CV

Перед відправкою CV перевір:

```text
[ ] CV має просту одноколонкову структуру.
[ ] Немає складних таблиць, інфографіки й прогрес-барів.
[ ] Розділи мають стандартні назви.
[ ] Навички написані текстом, а не тільки іконками.
[ ] Ключові слова з вакансії присутні у CV.
[ ] Ключові слова вбудовані в реальні bullet points.
[ ] Формулювання з вакансії частково дзеркаляться у Professional Summary, Skills або Experience.
[ ] PDF не ламає порядок читання тексту.
[ ] CV зрозуміле і людині, і машинному парсеру.
```

## 13. Головний принцип ATS-оптимізації

```text
Людина має побачити силу.
ATS має правильно прочитати структуру.
```

Тому CV має бути:

```text
просте
чітке
логічне
текстове
адаптоване під вакансію
```

Найкращий дизайн для технічного CV — це не той, який виглядає як постер, а той, який швидко й правильно читається рекрутером, hiring manager і ATS-системою.

## 14. Головний принцип

Сильне CV — це не хронологія життя.

Сильне CV — це професійна аргументація.

Воно має показувати:

```text
Я бачу проблему.
Я маю інструменти.
Я можу побудувати рішення.
Я створюю вимірювану або зрозумілу цінність.
```

Саме тому CV має переходити від стилю:

```text
I was doing...
```

до стилю:

```text
I built...
I engineered...
I validated...
I automated...
I designed...
I delivered...
```

## 15. Міні-висновок

Коли ви пишете CV, не думайте лише:

```text
Що я робив?
```

Думайте:

```text
Яку проблему я вирішував?
Які дані використовував?
Які інструменти застосовував?
Що стало краще після моєї роботи?
```

Це і є різниця між слабким списком обов'язків і сильним професійним резюме.
